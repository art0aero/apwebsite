# Website Testing Playbook

## Goal
Единый порядок запуска тестов для веб-сайтов, чтобы не терять шаги между средами и быстро повторять цикл: тест -> баг -> фикс -> ретест.

## Environments
- `Local static`: проверка HTML/CSS/JS и интеграций через браузерные e2e.
- `Stage/Prod`: только верификация интеграции и регрессии после деплоя.
- `Sandbox vs escalated`:
  - В sandbox часто блокируются сеть, localhost-порты и запуск браузера.
  - Для Playwright и сетевых API-проверок использовать режим с повышенными правами.

## Run Order (Required)
1. Preflight
- Проверить ветку: работа в `testing`.
- Проверить, что серверные функции/конфиги задеплоены и актуальны.
- Подготовить тестового пользователя и тестовые данные.

2. API Baseline
- Проверить базовые endpoint-ответы (`200/401/403` по ожидаемому сценарию).
- Для защищённых endpoints проверить валидный JWT поток.

3. Smoke UI
- Открытие страниц без runtime/page ошибок.
- Проверка навигации и видимости ключевых секций.

4. Core E2E Flow (Playwright)
- Логин/сессия.
- Основной пользовательский поток (например: тест -> сабмит -> результаты).
- Проверка критичных сетевых запросов и их статусов.

5. Security Checks
- XSS-проверка: данные из БД рендерятся как текст, без исполнения payload.
- Проверка, что чувствительная логика не осталась на клиенте.

6. Regression
- Повторный прогон после фиксов.
- Проверка консольных ошибок и ключевых метрик UI.

## Bug Loop (Mandatory)
1. Зафиксировать баг в отчёте:
- шаги воспроизведения,
- фактический результат,
- ожидаемый результат,
- где найдено (страница/API).

2. Исправить баг.
3. Повторно прогнать релевантные тесты + smoke.
4. Повторять цикл, пока все проверки не станут зелёными.

## Exit Criteria
- Все сценарии из матрицы PASS.
- Нет открытых P0/P1 дефектов.
- Отчёт тестирования заполнен (итерации, найденные баги, финальный статус).

## Default Artifacts
- `TEST_PLAN_AND_REPORT.md` — план, итерации, баги, итог.
- Playwright output JSON/лог (временный или в CI-артефакты).
- Короткий summary в PR/коммите.
